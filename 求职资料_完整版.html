1. 这个项目到底是什么（事实概述）

这是一个用 Spring Boot + Thymeleaf 搭建的个人作品集网站。它不只是一个好看的页面，而是把我的个人简介、技术栈、项目展示、GitHub 链接、项目演示视频、"Beyond the Code" 思考板块、工作经历、教育背景、联系方式集中在一个地方。

网站后端由一个 Spring 控制器把根路径 / 映射到一个 Thymeleaf 页面；项目展示内容则由前端 main.js 在运行时从结构化的 projects.json 读取并渲染成项目卡片，包含演示视频、SVG 架构/流程/ER 图、可展开的详情和 GitHub 链接。应用用 Dockerfile 容器化，端口从环境变量读取，因此可以部署到 Render 这类容器平台并绑定自定义域名。

它解决了什么问题

作为应届 MSc 毕业生找初级开发岗位，我需要一个可信、可自托管的统一入口来展示真实项目和我的工程思路，而不是只靠一份简历或一堆零散的 GitHub 仓库。这个网站把"我是谁、我会什么、我做过什么、我怎么思考"集中呈现给招聘方。

网站包含的主要板块（基于真实代码）

  • Hero 个人介绍 + 简历 PDF、邮箱复制、LinkedIn、GitHub 入口
  • README.md 风格的个人简介
  • Technical Toolkit 技术栈展示
  • Projects 项目展示（数据驱动，可展开详情、播放演示视频、预览图）
  • Experience 工作经历（China Telecom、Sainsbury's）
  • Education 教育背景（University of Glasgow、Hainan University）
  • Beyond the Code 四张可展开的思考卡片
  • Contact 联系方式（一键复制邮箱）

2. 技术架构与结构说明

Spring Boot + Thymeleaf

标准的 Maven Spring Boot 项目（Java 21，Spring Boot 2.7.10，依赖 web / thymeleaf / actuator）。一个 HomeController 把 / 映射到 Thymeleaf 的 index.html。服务端有意保持轻量。

静态资源（src/main/resources/static）

模块化 CSS（约 20 个文件）、一个 main.js，以及 images/（SVG 图）、videos/（MP4 演示）和被引用的简历文件。

项目数据来源

项目内容与页面解耦，存放在 data/projects.json。新增或修改项目只需改数据，不用动 HTML 标签。

JavaScript 交互

main.js 负责拉取 JSON 并渲染项目卡片，还处理卡片展开/收起、视频预览弹窗、图片/SVG 图预览、主题切换、点击复制邮箱等。

Docker / 部署支持

多阶段 Dockerfile（Maven 构建 → Temurin JRE 运行 JAR），端口由 server.port=${PORT:8081} 环境变量驱动，这正是它能在容器平台运行的原因。

项目数据是怎么组织的

一个 projects.json 数组，每个项目包含 id、title、summaryLine、stack、links.github、media.demoVideo、灵活的 details（sections[] 或 whyBuilt/architecture/database/diagrams[]）以及 featured 标记。目前定义了两个项目：JavaInsight 和 Job Application Tracker。

Projects 板块如何渲染

index.html 里只有一个空挂载点 #work-projects。main.js 拉取 JSON（cache: 'no-store'），按固定顺序排序，把项目拼成卡片 HTML（标签、GitHub 图标链接、演示视频预览、可展开详情），并绑定点击/键盘的展开收起、视频弹窗、SVG 规范化和图片预览弹窗。

Beyond the Code 如何工作

四张思考卡片直接写在 index.html 里；main.js 为它们加上无障碍的展开/收起行为（aria-expanded、一次只开一张、支持 Enter/Space 键）。

本地运行 / 部署

  • 本地：mvn -DskipTests clean package 然后 mvn spring-boot:run（默认端口 8081）。
  • 部署：通过 Dockerfile 构建并运行 JAR，端口由环境变量注入，可部署到 Render 等容器平台。

5. 面试问答准备

下面是这个项目可能引出的问题，以及简短的应答要点（中文思路，方便你转成英文表达）。

Q1. 你为什么要做这个网站？

找初级开发岗时需要一个集中、可自托管的地方展示真实项目和工程思路，用的是我工作中会用的 Java 技术栈。

Q2. 作品集为什么选 Spring Boot + Thymeleaf？

我想端到端地练习并展示一个偏后端的 Web 技术栈；一个控制器服务 Thymeleaf 页面，服务端刻意保持精简。要诚实承认后端很薄（单路由、无数据库）。

Q3. 项目数据是怎么组织的？

所有项目内容放在 projects.json（标题、简介、技术标签、GitHub 链接、演示视频、详情、图示），更新就是改数据而不是改 HTML。

Q4. Projects 板块怎么渲染的？

main.js 运行时拉取 JSON 并生成卡片，包含可展开详情、视频弹窗和 SVG 图预览。要清楚权衡：客户端渲染意味着内容依赖 JS，对 SEO 不友好、也没有服务端渲染。

Q5. JavaScript 交互是怎么实现的？

原生 JS 处理卡片展开/收起（支持键盘和 ARIA）、视频与图示弹窗、主题切换、复制到剪贴板；没有框架。可以提一下我解决的真实问题：SVG 的规范化（重置导出的 pan/zoom 变换、重算 viewBox）。

Q6. 怎么容器化并部署的？

多阶段 Dockerfile 先构建 JAR 再用 JRE 镜像运行；server.port=${PORT:8081} 让宿主注入端口，这就是它能跑在 Render 上的原因。要诚实说明 Render 和域名是在平台上配置的，仓库里没有这些配置文件。

Q7. 你怎么通过 GitHub 维护和更新？

内容改动就是对 projects.json 或模板做小修改并提交到 GitHub；展示的项目各自链接到它们自己的仓库。

Q8. 你会怎么改进它？

加自动化测试、把项目数据放到服务端接口、修正过时的 README、考虑对项目列表做服务端渲染以兼顾 SEO 和无 JS 回退。

  • 后端是刻意做薄的：只有一个路由服务一个页面，没有数据库、没有自建 REST 接口（只引入了 Actuator）、没有鉴权。不要把作品集网站说成复杂后端系统。

  • Technical Toolkit 里列的是技能，不等于这个仓库都用到了（比如 Redis、MySQL 并未在本仓库使用）。
  • Render 部署和自定义域名是平台侧的事实，仓库里没有 render.yaml / CNAME 等配置文件。可以在简历/面试里说，但要清楚它不是仓库里的产物。
  • README.md 已过时（它声称没有 Dockerfile，但其实有）。面试可主动提"我会修正它"。
  • 两个展示项目（JavaInsight、Job Application Tracker）是独立仓库，通过 GitHub 链接外链。
  • 不要夸大流量、性能、安全、可扩展性、SEO 或业务影响；也不要把它说成纯前端设计项目。
